Overview

 

Controlled access to tasks, activities and data within DevMan is limited to authorised personnel / permitted users.

 

•Define user roles and access functions for each role.

•Link one or more email address to user profiles.

•Audit stamps and trails provide transparency on data management and team usage.

•Change history can be viewed on selected important fields.

 

We recommend you assign a senior member of your team to care for this very important duty.

 

Tips and best practice

We strongly recommend that all users:

•Change their passwords regularly.

•Use a strong password with 8 or more characters (alphanumeric, preferably with caps and symbols.)

Features

 

       Access control

Limit who can access DevMan, or who can access certain components of the application.

Two types of access control available are:

•user-based

•role-based

The best or most appropriate solution for you depends on the specific needs of your organisation.

 

       Permissions - DevMan functions

Access rights:

•Activity      

Any activity based record (Has With, By and For links). This includes Interactions, reminders / actions due, letters and progress reports.

•Budget  

High level budget entries for organisations capturing payments transactions.

•Finance

Commitments, pledges, payments, donations, requisitions, remittances, tax certificates, payment processing, cashbook exports.

•Maintenance

Administration menu and field and flag settings.

•Person/Organisation  

Person and organisation profiles.

•Person/Organisation

Academic profiles.

•Project

Projects, bursaries, approaches, memberships.

•User  

User profiles.

       LDAP user authentication

LDAP user authentication is the process of validating a username and password combination with a directory server such MS Active Directory.

LDAP directories are standard technology for storing user, group and permission information and serving that to applications in the enterprise.

DevMan can be configured to use this, allowing users to login with their windows credentials.

 

       Inactive user accounts

If a user account has been inactive for three or more months:

•The account will automatically be disabled.

•When trying to log on, users will be notified that their account is inactive.

 

 

•Power users can reactivate the account.

oGo to Administration > Security > Users.

oSelect the user profile,  and then select the Reactivate option on the menu.

 

NOTE: This feature will be bypassed for users logging on with LDAP or AzureAD,

Please contact the support desk for more information or assistance.

 

 

 

 

How to set up a user

 

       User set up steps

1.Roles: define the various roles of your team. For example you may have Administrator, Project officer, Finance clerk, Fundraiser, Researcher, Director, etc.

2.Permissions: Create the role and set permissions per role.

3.User access: add a user profile and allocate a role.

4.Validate email address.

5.Send welcome email with log in details - userid and password.

 

       Create roles

Grant access rights to users with pre-defined role templates.

•To set up your various roles go to Administration > Security > Roles. Here you will see the roles already available.  

•To edit an existing role, select the relative line from the grid.  

•To add a new role select Add.

 

 

• Allocate a role name, e.g. Director, select Save and stay on page.

 

• Adjust the permissions for the different functions as required and select Save and stay on page / Save and close. (See Permissions - user access privilege  for an explanation)

 

• Add additional roles as needed.

 

       New user setup

 

       Add a person profile for the user

Add user details by creating a person profile and flagging the system status as for internal use.  Minimum required details include full name and email address.

 

       Add a user profile

Select Administration > Security > Users then select Add.

 

 

Add the user's full name. Use the inline search to link the user profile to the person profile.

Add a login name (set to the users first name).

Set the password (for the initial set up use the users first name).  Users are encouraged to change their password once they've logged in.

Set Multiple logins to 'prompt'.

Select Add.

 

 

If the user will send email from DevMan, an email identity must be added and the email address verified.

oSelect Add default email identity to send from the user's email address.

oAdditional from addresses can be added using the inline search in the email identity field.

•Select the applicable role for this user from the drop-down list.

•Save and stay on page.

You will be prompted to send an email verification request.

 

 

 

       Send welcome email

•Select Administration > Security > Users.

•Tick the check box/es of the users you want to send details to.

•Select Send > Communication > Email.

 

 

•In the template field type 'Welcome' and select the option from the drop-down menu.

•Select the applicable email stationery and change the subject line as needed.

•In the body of the email double click on the 'click here' - in the URL field add your DevMan system link then select OK.

•Select Send

 

 

User log and audit trail

Record audit trail:

•On any record, hover over the record stamp in the top right corner to view the history of record creation and updates.

 

 

•View change history for selected important fields on people and organisation profiles by selecting Maintenance > Change history.

 

 

•Review the top 10 user activity in the last 30 days on the Administration page user activity dashboard.

•Review when a user last logged in on the User list grid.

•Draw detailed audit reports from Administration > Security > Reports > Activity.
